## Findings

### F4-01 - useTerminal is a 897-line monolith with five orthogonal responsibilities
- Category: maintainability / decomposition
- Confidence: high
- Diff size: large (~900 LOC reorganized; no behavior change)
- Location: src/hooks/useTerminal.ts:90-897
- Today: One hook owns (a) Terminal/FitAddon construction L287-571, (b) WebGL/WebLinks/Unicode11/path-link addon wiring L154-278, (c) custom-key handler with 14 chord rules L411-535, (d) write queue + flush gate + visibility flush L116-117 L650-735, (e) clipboard paste blocker + OSC-0 rename L264-274 L388-403. Tests cover only the pure key-mapping helper.
- Proposed: Split into composable hooks driven from a thin useTerminal shell:
  - useXtermInstance({ scrollback, instanceKey }) -> { termRef, fitRef, ready, generation }
  - useXtermAddons(termRef, { enableWebgl, getCwd }) -> { webglRef }
  - useXtermAttachment(termRef, fitRef, { visible }) -> { attach, fit }
  - useXtermKeyboard(termRef, { onData }) using a key-shortcut table parameter
  - useTerminalWriteSink(termRef, { visible }) -> { write, writeBytes }
  Compose them in useTerminal returning the same public surface (attach, write, writeBytes, clear, focus, scroll helpers, ready, termGeneration).
- Side effects: TerminalPanel API unchanged. Each sub-hook becomes independently unit-testable with a mocked Terminal.

### F4-02 - terminalPathLinks holds an in-memory LRU cache despite "No caching" rule
- Category: rule violation / cache removal (user CLAUDE.md L6)
- Confidence: high
- Diff size: small
- Location: src/lib/terminalPathLinks.ts:35-60, 110-135
- Today: Map<string,string|null> capped 500 entries holds resolution verdicts, evicted on size and on cwd change. The clearCache callback returned from createPathLinkProvider is NEVER called by any consumer (useTerminal.ts:225 destructures { provider } only - clearCache is dead code). The cache also caches negative results so a directory tree change after the user creates a file means clicks silently fail until reload.
- Proposed: Delete the resolutionCache LRU and clearCache plumbing. Each provideLinks call is per-line and only fires on visible-line changes. Deduplicate within a single render cycle: build Set<string> of unresolved candidates from the current line, batch them into one resolve_paths IPC call, drop the result. Caching belongs server-side in path_resolve.rs (already TTL-cached per [TP-01]).
- Side effects: Extra IPC roundtrip for repeated-token lines; Rust side already caches the subtree index 60s. Removes a real correctness hazard (negative-cache never expires) and complies with no-caching rule.

### F4-03 - getBufferText walks the entire 100k-line scrollback per call; called inside a 500-iteration TUI scroll loop
- Category: performance (catastrophic worst case)
- Confidence: high
- Diff size: small
- Location: src/hooks/useTerminal.ts:862-876 (getBufferText); src/lib/tuiScrollSearch.ts:23,35,46,101 (callers via getSessionTranscript).
- Today: getBufferText translates every line of the active buffer (for i in 0..buf.length) and joins with \n. Buffer length is scrollback+viewport ~= 100,000 for Claude. scrollTuiToText calls getSessionTranscript(sessionId) once per Page-Up iteration, up to 500 iterations, plus down-scroll and recovery passes. Worst case: ~100k * 500 = 50M translateToString calls per single search.
- Proposed:
  1. tuiScrollSearch only needs the VIEWPORT for "is target visible?". Add getViewportText() that walks [buf.viewportY, buf.viewportY + term.rows) (<= ~80 lines) and use it in viewportIncludesTarget and scrollToTuiEdge's previous-viewport diff.
  2. Keep getBufferText for CommandPalette.tsx:47 (full-transcript clipboard dump) but make it lazy: return a Generator<string> rather than join up front.
- Side effects: Eliminates the 50M-iteration worst case. TUI page-scroll only changes the viewport, so correctness preserved. Edge-detect by viewport-equality remains valid (both samples now viewport-only).

### F4-07 - Path-link regex anchors and trailing-punct stripping have known false-positive shapes
- Category: correctness (robustness)
- Confidence: medium
- Diff size: small
- Location: src/lib/terminalPathLinks.ts:17-27
- Today: PATH_RE accepts:
  - bare multi-segment paths foo/bar/baz - matches ratios in prose (100/200/300), URL fragment tails, TUI Unicode "/" decorations.
  - bare filenames [\w\-]+\.[\w\-]{1,10} - matches e.g., i.e., version strings v1.2.3, TLD-shaped tokens (gov.au).
  - TRAILING_PUNCT_RE = /[).,;:"'>\]]+$/ trims trailing but not leading punct (("./foo") keeps the leading ("").
  - cleaned.includes("://") guard is post-trim; if "://" was in the trimmed tail it slips through.
- Proposed:
  1. Add leading-punct trim symmetric to trailing.
  2. Test "://" against untrimmed m[0].
  3. Tighten bare-filename: require >=1 alphanumeric on right of dot AND reject pure-numeric stems (1.2.3).
  4. Add unit tests src/lib/__tests__/terminalPathLinks.test.ts covering: e.g., i.e.,, 1.2.3, https://x.com/y.json,, ./foo/bar.tsx:42:10), C:\Users\foo\bar.txt, single-segment foo/bar.
  5. Move PATH_RE + trimTrailingPunct into a shared terminalPathRegex.ts util.
- Side effects: Reduces backend resolve_paths IPC volume on noisy lines and reduces bogus-link decoration in TUI prose.

### F4-08 - terminalRegistry uses module-scoped Maps with no provenance tracking; cleanup is opaque
- Category: maintainability / leak risk
- Confidence: high
- Diff size: small
- Location: src/lib/terminalRegistry.ts:5-31; consumers TerminalPanel.tsx:614-624.
- Today: Two module-scoped Maps. Cleanup is solely TerminalPanel's effect cleanup. If a session crashes mid-render, throws during unmount, or React StrictMode double-invokes, registries can hold stale Terminal references that prevent GC. Tests cover register/unregister API but not register-then-error scenarios.
- Proposed:
  1. Wrap each registry as a class TerminalRegistry { private map = new Map(); register/unregister/get/stats } with a clearAll(predicate) for test resets.
  2. Use WeakRef<Terminal>: registry transparently prunes dead refs on access.
  3. Move scrollBufferToText out of terminalRegistry.ts into tuiScrollSearch.ts (it's a search routine, not a registry method).
  4. Add an integration test that mounts/unmounts a TerminalPanel and asserts getRegisteredTerminalStats().length === 0 afterward.
- Side effects: WeakRef supported in Tauri webview (Chromium/WebKit). scrollBufferToText move is pure relocation.

### F4-11 - tuiScrollSearch uses MAX_SCROLLS=500 as an arbitrary delay-substitute
- Category: arbitrary delay (user CLAUDE.md L4)
- Confidence: medium
- Diff size: small
- Location: src/lib/tuiScrollSearch.ts:36, 90
- Today: MAX_SCROLLS = 500 caps the scroll loop. Combined with viewport-equality edge detection L48 and L111 (the deterministic termination), the 500 cap is dead unless a TUI reports a changing viewport for >500 PageUp keys without ever stabilizing. Cap hides genuine bugs where the viewport diff stops working.
- Proposed: Remove MAX_SCROLLS. Deterministic exit conditions are: (a) target found, (b) viewport === prevViewport (edge), (c) signal.aborted. If a hard cap is required, derive from term.buffer.active.length (<= scrollback) rather than 500.
- Side effects: Possibly longer search times in pathological TUIs; matches no-arbitrary-numbers rule.

### F4-12 - waitForRender(sessionId, timeoutMs = 120) is an arbitrary delay
- Category: arbitrary delay (user CLAUDE.md L4)
- Confidence: high
- Diff size: small
- Location: src/lib/terminalRegistry.ts:57-73
- Today: Resolves on first term.onRender event OR after 120ms. The 120ms isn't documented or tunable, exists as a defense against TUIs that don't fire onRender after a key. But that's exactly what viewportIncludesTarget after the wait should catch - if no render fires, viewport hasn't changed, edge-detect terminates the loop.
- Proposed: Remove setTimeout fallback. If onRender doesn't fire, the loop terminates via edge-detect since viewport string is unchanged. Test environments mock onRender to fire synchronously (existing test already does this with _fireRender()).
- Side effects: Aligns with no-arbitrary-delays rule.

### F4-15 - Recursive flushWriteQueue can synchronously stack-overflow on a fully-buffered hidden tab
- Category: latent bug
- Confidence: medium
- Diff size: small
- Location: src/hooks/useTerminal.ts:705-722
- Today: term.write(batch.data, callback) can invoke its callback synchronously when the parser finishes a small batch. Callback calls flushWriteQueue which may take another batch and call term.write again. 256KB merge cap bounds recursion at total_bytes/256KB - for a 50MB queue ~200 frames - unlikely to overflow but possible.
- Proposed: queueMicrotask(flushWriteQueue) instead of direct call. Same throughput, no recursion.
- Side effects: One extra microtask per batch during a flush. Negligible; protects against a class of stack-blowup bugs.
