## Findings

### F10-1: Icons.tsx 45 named exports, no shaking benefit at this site count

- **Category**: code-shape / refactor
- **Confidence**: medium (good DX, low real cost in production)
- **Diff size**: large (touches 12 import sites)
- **Location**: `src/components/Icons/Icons.tsx`, plus 12 import sites under `src/components/**` and `src/App.tsx`
- **Today**: 45 named exports of `IconStop`, `IconClose`, `IconReturn`, etc. Imported individually (e.g. `import { IconStop, IconClose, IconReturn, IconGear, IconFolder } from "./components/Icons/Icons";` in `App.tsx:30`). 46 distinct icon names appear in source, 12 files import from Icons.
- **Proposed**: Two viable options, in order of conservatism:
  - **(a) String-keyed `<Icon name="stop" size={16} />` registry.** Replace 45 exports with a single `ICON_PATHS: Record<string, ReactNode>` map and one `<Icon>` component. Reduces 495 → ~150 LOC. Tradeoff: loses TypeScript autocomplete on icon name (could be recovered with `keyof typeof ICON_PATHS`).
  - **(b) Keep named exports but make the `Icon` wrapper accept an `as` prop** so dynamic lookup is possible without losing per-import typing. Smaller diff.
- **Side effects**: 12 file edits for option (a). Tree-shaking is irrelevant in a Tauri app (single bundle, no code-splitting), so the "DCE saves bundle" argument is weak — pick on DX only.
- **Decision needed**: this is largely cosmetic. **Recommend NOT doing now** — 12-site touch for cosmetic gain, and the current shape is consistent and readable. Revisit if more icons are added (>60).

### F10-2: DebugPanel 500ms polling instead of subscription

- **Category**: performance / event model
- **Confidence**: high
- **Diff size**: small
- **Location**: `src/components/DebugPanel/DebugPanel.tsx:79-101`, `src/lib/debugLog.ts:43-46,229,237`
- **Today**: `setInterval(... , 500)` polls `getDebugLogGeneration()`. If the generation hasn't moved, no work; if it has, the panel re-fetches a snapshot. The store already increments `generation++` on every `pushEntry` and `clearDebugLog()`, but exposes no subscription API.
- **Proposed**: Add `subscribeDebugLog(listener: () => void): () => void` to `debugLog.ts`. Maintain a `Set<() => void>` of listeners; call them after `generation++`. DebugPanel calls `subscribeDebugLog(forceUpdate)` and drops the interval. Throttling/coalescing happens via React's `useSyncExternalStore` + `requestIdleCallback` if needed (no fixed timers).
- **Side effects**: Removes the 500ms idle CPU cost when DebugPanel is open with no log activity. Still gates updates on text-selection (preserve user-highlighted ranges) by deferring inside the listener instead of returning early in the interval.
- **Note**: this is a real improvement, not a phantom one — the polling interval is unconditional and runs even when the panel is hidden if it's mounted. Verify mount lifecycle (`grep -n DebugPanel src/components/RightPanel/`) to confirm; if it unmounts when collapsed, the win is smaller.

### F10-3: SearchPanel debounce uses `setTimeout`, dependency on stable `executeSearch` is fragile

- **Category**: bug-bait / hooks
- **Confidence**: medium-high
- **Diff size**: small
- **Location**: `src/components/SearchPanel/SearchPanel.tsx:148-154`
- **Today**:
  ```
  useEffect(() => {
    if (debounceRef.current) clearTimeout(debounceRef.current);
    debounceRef.current = setTimeout(executeSearch, DEBOUNCE_MS);
    return () => { if (debounceRef.current) clearTimeout(debounceRef.current); };
  }, [executeSearch]);
  ```
  `executeSearch` is `useCallback` over `[query, caseSensitive, useRegex, searchableSessionScope]`. The eslint-disable on line 144 hints the author already knew this. Every keystroke remounts the timer.
- **Proposed**: Extract a `useDebouncedEffect(fn, deps, ms)` hook (or use `useDebouncedCallback`). The current shape works but is hand-rolled at each call site. Bigger ask: collapse the 7 `useState` into a single `useReducer`-style query state to keep dispatch coherent (`{query, caseSensitive, useRegex, results, activeIndex, regexError, scrolling, navigationMessage}`), which also makes "clear" a single dispatch instead of 7 setters.
- **Side effects**: Lower; mostly a readability/correctness refactor. The debounce-as-state-derived-effect pattern is not a "delay" in the sense the user CLAUDE.md prohibits — debouncing user input is the legitimate exception.

### F10-4: SearchPanel — abort guarantee on unmount only covers the scroll, not the IPC

- **Category**: bug-bait / async
- **Confidence**: high
- **Diff size**: tiny
- **Location**: `src/components/SearchPanel/SearchPanel.tsx:108-143,304-309`
- **Today**: `searchGenRef` is the in-flight discriminator for `invoke<JsonlMatch[]>("search_jsonl_files", ...)`. On unmount, only `abortRef.current?.abort()` runs; the search invoke is not aborted, only its result is discarded by the gen guard. That's fine for stale-result handling, but if the IPC is slow, the `setResults`/`setActiveIndex` calls in the catch path can fire after unmount (they're already gen-guarded, so React will warn, not corrupt state).
- **Proposed**: Bump `searchGenRef` in the unmount cleanup (currently it's only bumped on `handleClear`). Two-line fix in the existing unmount-cleanup useEffect.
- **Side effects**: Eliminates "set state after unmount" warnings under slow-disk conditions. No behavioral change on the happy path.

### F10-5: SearchPanel — `<input onKeyDown>` registered on the panel `div`, not the input

- **Category**: bug
- **Confidence**: medium
- **Diff size**: tiny
- **Location**: `src/components/SearchPanel/SearchPanel.tsx:312`
- **Today**: `<div className="search-panel" onKeyDown={handleKeyDown}>` — the handler relies on bubble phase reaching the panel. `e.preventDefault()` on `Enter`/`F3`/`Ctrl+L` works inside the input because the event bubbles. But `Escape` is not handled here, and for `F3` (a global "Find next" semantic) the current scope means the user must have focus inside the search panel. Not a real bug, just a scope mismatch with the typical `F3` UX.
- **Proposed**: Move keydown to the `<input>` itself for `Enter` and `Ctrl+L` (those are textbox semantics). Add a `window` listener for `F3` if global "find next" is desired. Otherwise leave alone — current behavior is fine if F3 is intended to be panel-scoped.

### F10-6: CommandBar — `writeToPty` on click without coalescing

- **Category**: ux / robustness
- **Confidence**: medium
- **Diff size**: tiny
- **Location**: `src/components/CommandBar/CommandBar.tsx:64-77,138-148`
- **Today**: A double-click on a slash pill writes the command twice (once per click). For Ctrl+Click sends, this means two `\r` to the PTY → two command submissions. For plain click ("type into terminal without sending"), two clicks types the command twice — usually visible and recoverable, but a footgun. The brief flagged "no debouncing → rapid clicks could send duplicates" — the user CLAUDE.md explicitly permits a click debounce.
- **Proposed**: Wrap `handleClick` in a `useRef<number>(0)` timestamp guard with a 250ms threshold. On call: `if (Date.now() - lastClickTs.current < 250) return; lastClickTs.current = Date.now();`. Apply the same to the history-strip pill (line 142). No timer, no setTimeout — a logical guard, not a delay.
- **Side effects**: Eliminates accidental duplicate submission. Affects keyboard repeat (Space-and-hold on a focused button) but that's the desired prevention.
- **Stronger alternative**: track per-pill `inFlight` boolean via the session's command-history echo (the PTY-roundtrip TAP event clears it). More work, more correct.
