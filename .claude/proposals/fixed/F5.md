## Findings

### F5-1 — `gitRepoCache` violates the user CLAUDE.md "no caching" directive
- **Category**: rule violation / design
- **Confidence**: high
- **Location**: useTapEventProcessor.ts:52
- **Today**: Module-scoped `Map<string, boolean>` memoises `git_repo_check` results forever per `workingDir`. No invalidation. User who runs `git init` in a previously-checked directory never sees new behaviour without app restart.
- **Proposed**: Move check to a single `git_list_changes` call where Rust reports `is_repo: false` cheaply, OR delete the cache entirely (~1-2ms over IPC; idle hysteresis fires every ~2s). Recommend deletion. **Ask user first.**

### F5-2 — `dispatchBatch` is not actually atomic — invariant violated
- **Category**: correctness (latent)
- **Confidence**: high
- **Diff**: trivial
- **Location**: tapEventBus.ts:37-45
- **Today**: Documented as atomic but implementation is handler-major, event-minor — handler A receives all events first, then handler B. Bug dormant today (only one subscriber) but breaks future second subscribers.
- **Proposed**: Swap loops to event-major OR rewrite the comment. Swap.

### F5-3 — `apiIpResolvedRef` global guard is per-hook-instance, not global
- **Category**: dead-code
- **Confidence**: high
- **Diff**: trivial
- **Location**: useTapEventProcessor.ts:298, 878-882
- **Today**: Each session's processor triggers one `resolve_api_host` call on first ApiFetch. Cross-session gate is `!useSettingsStore.getState().apiIp` (works correctly). Ref is dead weight.
- **Proposed**: Delete ref. Keep store check only.

### F5-4 — Throttle in `useTapEventProcessor` is "min-delay" not real throttle
- **Category**: correctness / CLAUDE.md compliance
- **Confidence**: high
- **Diff**: small
- **Location**: useTapEventProcessor.ts:917-928
- **Today**: `Math.max(VALIDATE_THROTTLE_MS - elapsed, 200)`. If activity arrives every 500ms, fires every 200ms (the floor), not every 1500ms. Wrong implementation.
- **Proposed**: Standard trailing-edge throttle: `const next = Math.max(lastValidate + VALIDATE_THROTTLE_MS - Date.now(), 0);`.

### F5-5 — `TapMetadataAccumulator.diff()` is JSON-fingerprint based — wasteful but tolerable
- **Category**: performance
- **Confidence**: medium
- **Diff**: small
- **Location**: tapMetadataAccumulator.ts:596
- **Today**: `JSON.stringify(metadata)` runs on every event to detect changes. Accumulator already knows when fields change because every mutation is in a `case` branch.
- **Proposed**: "Dirty bit" flipped in each case (`dirty = true`). `diff()` returns early if not dirty.

### F5-6 — `TurnDuration → idle` fallback documents a known race using 2s arbitrary delay
- **Category**: correctness / CLAUDE.md compliance
- **Confidence**: high
- **Diff**: small
- **Location**: tapStateReducer.ts:102-113
- **Today**: Comment says "subagent's TurnDuration could cause a transient false idle. … the main agent's next TurnStart immediately corrects it, and tab flash debounce (2s) suppresses the UI notification." This is **explicitly a 2-second arbitrary delay used as control flow** — exactly what CLAUDE.md rule 4 forbids.
- **Proposed**: Add `agentId` (or `sidechain` marker) to TurnDuration event in classifier; only treat as idle when from main agent. If CLI doesn't emit, drop TurnDuration → idle entirely.

- **Confidence**: medium

- **Location**: tapSubagentTracker.ts:184
- **Proposed**: Named constant or table entry alongside noisy-event-kinds catalog.

### F5-8 — tapClassifier table-dispatch in TWO places (cat dispatch + parse classifier)
- **Category**: design / aggressive simplification
- **Confidence**: high
- **Diff**: medium
- **Location**: tapClassifier.ts:968-1325 (cat dispatch, 32 branches), :246-307 (parse, 6 branches)
- **Today**: Three sub-classifiers; two dispatch by `cat` or `parsed.type` and are clean candidates for table dispatch.
- **Proposed**: `Record<string, (ts, entry) => TapEvent | null>` registries. ~200 LOC removed.

### F5-9 — classifyStringify is NOT a clean table dispatch — but ordered predicate-matchers help
- **Category**: design / aggressive simplification
- **Confidence**: high
- **Diff**: medium
- **Location**: tapClassifier.ts:312-891 (36 branches)
- **Today**: Each branch is a *shape predicate*; order matters (comments say "must come BEFORE ModeChange").
- **Proposed**: `Array<{ test: (p: any) => boolean; build: (ts: number, p: any) => TapEvent }>` — ordering becomes load-bearing data, not "must come BEFORE" comments. ~30% LOC reduction.

### F5-10 — useTapEventProcessor split by concern (~951 → 6×~150)
- **Category**: refactor / cohesion
- **Confidence**: high
- **Diff**: large
- **Today**: Bundles 8 concerns: state reducer, metadata accumulator, subagent tracker, activity-store file tracking, worktree cwd detection, codex auto-rename, SystemPromptCapture bridge, settled-idle git scan + path validation.
- **Proposed**: Split into:
  - `useTapEventProcessor.ts` — orchestrator + concerns 1-3 (~250)
  - `useTapActivityTracker.ts` — concern 4 (~250)
  - `useTapWorktreeSync.ts` — concern 5 (~80)
  - `useTapCodexNaming.ts` — concern 6 (~60)
  - `useTapPromptCaptureBridge.ts` — concern 7 (~80)
  - `useTapSettledIdleHandler.ts` — concern 8

### F5-11 — useTapPipeline FLUSH_INTERVAL_MS=2000 is a cadence, fine
- **Category**: clarification
- **Confidence**: high
- **Today**: 2-second flush is disk-write batching, not control flow. Bound IPC rate, preserve real-time classify→dispatch (synchronous on every payload). FLUSH_THRESHOLD=50 already early-flushes under load.
- **Proposed**: No change. Recursive `setTimeout` could become `setInterval` for cleaner cleanup.

### F5-12 — Add a golden-file integration test
- **Category**: test coverage
- **Confidence**: high
- **Diff**: small
- **Today**: 234KB of unit tests, but no integration test exercising full pipeline against recorded session.
- **Proposed**: One golden-file test per CLI (claude, codex). Feeds recorded `tap-entries.jsonl` through `classifyTapEntry → reducer + accumulator + tracker` and snapshots final state. Catches every classifier-reorder bug at once.

## Cross-cutting concerns

- F5-1 (gitRepoCache) is the only direct CLAUDE.md rule violation in the unit. Highest priority to remove pending user confirmation.
- F5-6 documents a tab-flash 2s debounce as load-bearing — this is fragile and should be eliminated by adding `sidechain` marker, not preserved.
- F5-10 split is the largest refactor; lowest risk because each split-out hook subscribes to the same bus and ordering is preserved by registration order.
